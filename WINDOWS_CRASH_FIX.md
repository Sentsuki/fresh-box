# Windows 后台运行崩溃问题修复

## 问题分析

你的应用在Windows后台运行时异常退出的主要原因包括：

### 1. Mutex锁死锁问题
- 原代码使用 `lock().unwrap()` 可能导致panic
- 在高并发或异常情况下会使应用崩溃

### 2. 托盘图标初始化失败
- `app.default_window_icon().unwrap()` 在图标不存在时会panic

### 3. 缺少异常处理
- 没有全局panic hook来捕获崩溃信息
- 进程管理不够健壮

## 修复内容

### 1. 统一的进程管理机制
- 应用启动时自动检测系统中已运行的sing-box进程并接管管理
- 不再区分"内部"和"外部"进程，统一管理所有sing-box进程
- 通过PID跟踪确保进程状态一致性，避免重复启动

### 2. 智能进程接管
- 应用重启后能自动检测并接管现有的sing-box进程
- 使用三层管理机制：直接管理、PID管理、自动发现
- 确保应用意外退出后重新打开仍能正常管理sing-box

### 3. 新增的命令
- `initialize_singbox_state` - 初始化时检测现有进程
- `refresh_singbox_detection` - 手动刷新进程检测状态
- `get_singbox_status` - 获取详细的运行状态信息

### 4. 添加了全局panic hook
- 在 `main.rs` 中添加了panic处理
- 崩溃时会生成 `crash.log` 文件
- 包含时间戳和详细错误信息

### 2. 改进了Mutex锁处理
- 所有 `lock().unwrap()` 都改为安全的错误恢复
- 使用 `poisoned.into_inner()` 从中毒的锁中恢复

### 3. 安全的托盘图标处理
- 检查图标是否存在再设置
- 避免因图标问题导致的崩溃

### 4. 添加了健康检查功能
- 新增 `health_check_singbox` 命令
- 可以检查sing-box进程状态
- 窗口获得焦点时自动检查

### 5. 添加了依赖
- 添加了 `chrono` 用于时间戳记录

## 使用方法

### 1. 重新编译应用
```bash
cd src-tauri
cargo build --release
```

### 2. 使用调试脚本
运行 `debug_start.bat` 来启动应用并查看详细信息：
- 会显示之前的崩溃日志
- 启动后检查进程状态
- 设置详细的日志级别

### 3. 使用监控脚本
运行 `monitor.bat` 来持续监控应用状态：
- 每30秒检查一次进程状态
- 实时显示时间戳
- 检测到崩溃时显示日志

### 4. 查看崩溃日志
如果应用崩溃，检查以下文件：
- `crash.log` - 应用崩溃日志
- `bin/singbox.log` - sing-box运行日志

## 额外建议

### 1. 系统兼容性
- 确保Windows Defender没有误杀应用
- 检查防火墙设置
- 考虑以管理员权限运行

### 2. 资源监控
- 监控内存使用情况
- 检查磁盘空间
- 观察CPU使用率

### 3. 网络问题
- 检查sing-box配置是否正确
- 确认网络连接稳定
- 查看代理设置

### 4. 系统事件
- 检查Windows事件查看器
- 关注系统休眠/唤醒事件
- 监控电源管理设置
